# JWT Implementation Plan

## Goal

Replace the current unsigned `session_id` cookie with a signed JWT-based session while keeping the existing SvelteKit remote-function authorization model intact.

## Current State

- `src/lib/server/auth.ts` reads `cookies.get('session_id')` and treats the value as a user ID.
- `getUser()` then loads the user row from MySQL via `selectUsers(...)`.
- `requireUser()` and `requireRole()` are already used consistently across protected `data.remote.ts` files.
- `src/hooks.server.ts` sets `event.locals.user = await getUser()` for every request.
- `src/routes/auth/data.remote.ts` logs in with email/password and calls `setSession(user.id)`.

## Recommended Approach

Use a single signed JWT stored in an `httpOnly` cookie.

- Keep the cookie transport model because the app is server-rendered and already depends on cookies.
- Keep the DB lookup after token verification so role changes, linked student/lecturer changes, and deleted users are reflected immediately.
- Put only stable identity data in the token, ideally just `sub` (user ID) plus standard metadata such as `iat` and `exp`.
- Do not introduce refresh tokens in this phase. The current app has no separate public API or mobile client flow that needs bearer-token rotation.

This is the smallest secure migration from the current implementation.

## Implementation Steps

1. Add JWT support.

- Add a JWT library compatible with the current ESM/SvelteKit setup. `jose` is the best fit here.
- Add a private env var for the signing key, e.g. `JWT_SECRET`.
- Optionally add `JWT_ISSUER` and `JWT_EXPIRES_IN`, but a hardcoded 7-day expiry matching the current cookie lifetime is acceptable for the first pass.

2. Update `src/lib/server/auth.ts`.

- Introduce token helpers inside this file unless it becomes too large:
  - `signSessionToken(userId: string)`
  - `verifySessionToken(token: string)`
- Change `setSession(userId)` so it signs a JWT and stores that token in the cookie.
- Change `getUser()` so it:
  - reads the cookie
  - verifies the JWT signature and expiry
  - extracts `sub`
  - queries `selectUsers(...)` with that user ID
  - returns `null` for invalid, expired, or missing tokens
- When verification fails, clear the cookie so bad tokens do not persist.
- Keep `requireUser()` and `requireRole()` unchanged from a caller perspective.

3. Keep the request lifecycle stable.

- `src/hooks.server.ts` should continue calling `getUser()` exactly once per request.
- `src/app.d.ts` likely does not need structural changes because `App.Locals.user` can stay as-is.

4. Keep the auth entry points stable.

- `src/routes/auth/data.remote.ts` should continue using `login(...)`, `setSession(...)`, `clearSession()`, and `getCurrentUser().refresh()`.
- No client-side JWT parsing should be introduced.
- No `Authorization: Bearer` flow is needed in this phase.

5. Decide on cookie naming.

- Preferred: rename `session_id` to something semantic like `session_token` or `auth_token`.
- Minimal alternative: keep `session_id` and change only its value format.
- Recommended choice: rename it to avoid ambiguity and make the migration explicit.

6. Verify env and deployment expectations.

- Ensure the runtime environment used by `adapter-node` supplies `JWT_SECRET`.
- Reuse the existing `secure`, `sameSite`, `httpOnly`, `path`, and `maxAge` cookie settings.
- If the app runs behind HTTPS in production, keep `secure: url.protocol === 'https:'` or switch to a stricter production-aware check if needed later.

## Files Likely Touched During Implementation

- `package.json`
- `package-lock.json`
- `src/lib/server/auth.ts`
- `src/routes/auth/data.remote.ts` if the cookie API signature changes
- `src/hooks.server.ts` only if small cleanup is needed
- Optional env documentation file if the repo maintains one later

## Verification Plan

1. Static validation.

- Run `npm run check`.

2. Manual auth verification through the existing `/test` page.

- Log in with a valid account and confirm `getCurrentUser` returns the expected user.
- Refresh the page and confirm the session persists.
- Log out and confirm protected remote functions return unauthorized again.

3. Negative-path verification.

- Tamper with the auth cookie and confirm the request is treated as unauthenticated.
- Confirm an expired or invalid token is cleared and does not resolve to any user.
- Confirm a deleted user linked to a still-valid token resolves to `null`.

## Non-Goals For This Phase

- Refresh-token rotation
- Client-readable tokens
- Third-party identity providers
- Role/permission claims as the sole source of authorization truth
- Public API bearer-token support

## Risks And Notes

- The current implementation is vulnerable because a user ID in a cookie can be forged; JWT signing fixes that immediately.
- JWT does not remove the DB lookup in this plan by design. That is intentional to preserve current authorization behavior.
- If instant server-side logout across all devices is required later, JWT alone is not enough; that would need token revocation, a version field, or a session table.

## Suggested Execution Order

1. Add dependency and env requirement.
2. Refactor `auth.ts` to sign and verify JWT cookies.
3. Keep hooks and remote auth flow compatible.
4. Run type checks.
5. Manually validate login, persistence, logout, and tamper resistance.
